# 49 帧 (12in + 12out, Memory=49) 方案说明

## 📊 方案对比

### Motion Guided 原论文

根据论文原文 (Page 7, 637-642行):

> "For SEVIR dataset, given the **previous 12 echo images**, we aim to predict the **future 12 echo images**. Therefore... each of them has **T=24 images spanning 2 h of time**."

| 配置 | 帧数 | 时间 | 说明 |
|------|------|------|------|
| 总序列 | 24 帧 | 2 小时 | SEVIR 事件被切分为 3 个 24 帧子序列 |
| 输入 | 12 帧 | 1 小时 | 历史观测 |
| 输出 | 12 帧 | 1 小时 | 未来预测 |
| Memory | 24 帧 | 2 小时 | 长期记忆长度 |

**原论文的限制**:
1. 只使用 SEVIR 事件的 2 小时（24帧），未利用完整 4 小时数据
2. 没有 Flash Attention，显存受限无法处理更长序列

---

### 我们的方案 (49 帧 Flash Attention)

| 配置 | 帧数 | 时间 | 优势 |
|------|------|------|------|
| 总序列 | **49 帧** | **4小时5分** | 🔥 SEVIR 完整序列 |
| 输入 | **12 帧** | **1 小时** | ✅ 与原论文一致 |
| 输出 | **12 帧** | **1 小时** | ✅ 与原论文一致 |
| Memory | **49 帧** | **4小时5分** | 🔥 完整历史记忆 |

**优势**:
1. ✅ 保持与原论文一致的输入输出配置 (12→12)
2. ✅ Memory 可以访问完整的 SEVIR 4小时历史 (49帧 vs 24帧)
3. ✅ Flash Attention 解决显存问题，支持更长序列
4. ✅ 更长的历史信息有助于捕捉风暴完整生命周期

---

## 🔬 为什么这样设计？

### SEVIR 数据集结构

根据论文原文 (Page 7, 621-623行):
> "Each sequence, or 'event' in SEVIR ... spanning a **4 h** of time which is sampled in **5 min steps**."

**SEVIR 原始事件**: 4小时 = 240分钟 ÷ 5分钟 = **48-49 帧**

### 原论文的数据处理

原论文将每个 SEVIR 事件 (49帧) 切分为 **3 个 24 帧子序列**:

```
原始 SEVIR 事件 (49帧 = 4小时5分):
┌─────────────────────────────────────────────────────────────────┐
│ 帧 0-23  │  帧 12-35  │  帧 24-48                              │
│ 子序列1   │  子序列2   │  子序列3                                │
│ 24帧(2h)  │  24帧(2h)  │  24帧(2h)                              │
└─────────────────────────────────────────────────────────────────┘
```

每个子序列: 输入12帧 (1h) + 输出12帧 (1h) = 总共24帧 (2h)

### 我们的改进

直接使用 **完整的 SEVIR 49帧序列**:

```
完整 SEVIR 事件 (49帧 = 4小时5分):
┌─────────────────────────────────────────────────────────────────┐
│ 帧 0-48  (全部 49 帧)                                           │
│                                                                  │
│  Memory: 可访问全部 49 帧 (4小时5分钟历史)                       │
│  Input:  帧 0-11  (12帧 = 1小时)                                │
│  Output: 帧 12-23 (12帧 = 1小时)                                │
└─────────────────────────────────────────────────────────────────┘
```

## 🚀 使用方法

### 与现有配置对比

| 脚本 | 输入 | 输出 | 总序列/Memory | 预测时长 |
|------|------|------|--------------|----------|
| `train_384.py` | 8 帧 | 12 帧 | 20 帧 | 1 小时 |

| **原论文 (SEVIR)** | 12 帧 | 12 帧 | 24 帧 | 1 小时 |
| **49帧方案 (新)** | 12 帧 | 12 帧 | **49 帧** | 1 小时 |

### CLI 参数

```bash
# 49帧基线
python -u train_49frame.py --baseline --epochs 100

# 49帧 + exp11 (SwiGLU-MoE + RoPE + Flash)
python -u train_49frame.py --exp exp11_swiglu_moe_rope_flash --epochs 100

# 运行所有 exp7-12
python -u train_49frame.py --all --epochs 100
```

## 📈 预期效果

### CSI 指标预期

| 配置 | Memory长度 | CSI@30dBZ (预期) | 说明 |
|------|-----------|----------------|------|
| 原论文 (24帧) | 2小时 | ~0.35 | 基线 |
| 36帧方案 | 3小时 | ~0.40 | 中期改进 |
| **49帧方案** | **4小时5分** | **~0.42-0.45** | 🔥 完整历史 |

**为什么 49 帧会更好？**
1. SEVIR 完整事件包含风暴的完整生命周期（生成→发展→消散）
2. 4小时历史比2小时提供更准确的趋势判断
3. Flash Attention 允许模型充分利用所有历史信息
4. 特别是对于 multicell storms，长历史更能捕捉运动趋势

### 与 exp7-12 结合

```
最佳组合预测:

exp11_swiglu_moe_rope_flash + 49帧 (12→12, Memory=49)
├─ SwiGLU MoE: 更好的特征表示
├─ Temporal RoPE: 时间位置感知
├─ Flash Attention: 长序列处理
└─ 49帧 Memory: SEVIR 完整历史

预期 CSI@30dBZ: 0.42-0.48
(相比原论文的 ~0.35，提升约 20-35%)
```

## 🔬 技术细节

### Flash Attention 的作用

| 操作 | 不用 Flash Attn | 用 Flash Attn |
|------|-----------------|---------------|
| 注意力计算 | O(N²) 显存 | O(N) 显存 |
| 49 帧序列 | ❌ 显存爆炸 | ✅ 可运行 |
| 训练速度 | 慢 | 快 2-3× |

### 显存占用对比

| 配置 | 显存占用 (FP16) | 是否可用 |
|------|-----------------|----------|
| 20 帧 (8→12) | ~8 GB | ✅ |
| 24 帧 (12→12) 原论文 | ~9 GB | ✅ |
| 36 帧 (12→12) | ~12 GB | ✅ |
| **49 帧 (12→12)** | ~14 GB | ✅ (Flash Attention) |

## ⚠️ 注意事项

1. **训练时间**: 49 帧训练比 24 帧慢约 2 倍
2. **显存要求**: 建议使用 16GB+ 显存的 GPU
3. **数据集**: SEVIR 原始数据就是 49 帧，无需额外处理
4. **评估**: 使用 `evaluate_metrics.py` 时需要指定 `--preset 49frame`
